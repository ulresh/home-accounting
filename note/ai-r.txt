## Коротко

1. **Инкрементно из сети — частично да.** Есть API `Json.decodeFromStream(InputStream)`, который читает поток **порциями по мере поступления** (не загружая всё в память). Но модель — **блокирующая «pull»**: библиотека сама *вытягивает* байты из `InputStream`, а вы **не можете «толкать» (push) в неё свои буферы** по мере прихода из сокета. Асинхронного/неблокирующего (suspend, NIO, корутины) streaming-входа нет.
2. **Конец JSON-конструкции — да, находит сама.** Парсер определяет границу значения по синтаксису (баланс `{}`/`[]`, закрытие строк), внешний `Content-Length` или разделитель не нужен.

## Подробнее по п.1 — инкрементность

Что есть (JVM, помечено `@ExperimentalSerializationApi`):

```kotlin
// читает из InputStream порциями, низкий расход памяти,
// возвращает результат только когда значение полностью разобрано
val obj = Json.decodeFromStream<MyType>(inputStream)
```

Важные ограничения:

- **Pull, а не push.** Вы передаёте `InputStream` (например, `socket.getInputStream()`), и библиотека сама блокирующе читает из него столько, сколько нужно. Скормить ей вручную «вот пришёл очередной `ByteArray` из сети» — нельзя. Если у вас неблокирующий источник (Netty/NIO/ktor `ByteReadChannel`), его нужно обернуть в блокирующий `InputStream`-адаптер.
- **Нет частичного результата.** Пока верхнее значение не дочитано до конца, объекта вы не получите. SAX-подобного «событие на каждый токен по мере прихода» **публичного API нет** — внутренний лексер/ридер не выставлен наружу (в отличие от Jackson `JsonParser` или Gson `JsonReader`).
- **Память** при этом расходуется экономно: `decodeFromStream` декодирует напрямую в объект, не строя полное дерево `JsonElement`, и читает поток сегментами, отбрасывая разобранное. То есть «инкрементно по буферам» в смысле чтения — да; «инкрементно по выдаче результата» — нет.

## Подробнее по п.2 — поиск конца конструкции

Да, это встроено. Парсер дочитывает поток ровно до конца синтаксически завершённого значения и останавливается. На этом построен второй API — разбор **нескольких** JSON-значений из одного потока:

```kotlin
// лениво отдаёт по одному значению, сам определяя границы между ними
val seq: Sequence<MyType> = Json.decodeToSequence<MyType>(inputStream)
for (item in seq) { /* ... */ }
```

Режимы определения границ (`DecodeSequenceMode`):
- `WHITESPACE_SEPARATED` — значения подряд через пробелы/переводы строк (NDJSON / JSON-lines: `{...}\n{...}\n{...}`);
- `ARRAY_WRAPPED` — элементы внутри одного большого `[ ... ]`, отдаются по одному, не держа весь массив в памяти;
- `AUTO_DETECT` (по умолчанию) — определяет сам по первому непустому символу.

Здесь как раз видно, что библиотека **сама находит конец каждой конструкции** — иначе она не смогла бы делить поток на отдельные значения.

## Практический вывод

- Нужно «отдать сокет и пусть само вытянет один JSON и поймёт, где он кончился» → **`decodeFromStream` / `decodeToSequence` подходят** (это, например, и делает Ktor-клиент с kotlinx.serialization).
- Нужен **неблокирующий push-парсинг с backpressure** (кормить partial-буферами, получать токены/частичные результаты) → **только kotlinx.serialization не хватит**; берите Jackson (`JsonParser`, `non-blocking` async-парсер `JsonFactory` + `ByteBufferFeeder`) или комбинируйте kotlinx.serialization с обёрткой блокирующего потока поверх вашего async-канала.

Если уточните, какой у вас источник (raw-сокет, Ktor `ByteReadChannel`, Netty) — подскажу конкретную обвязку под `decodeFromStream`.

Да, это возможно — ключ в том, чтобы читать следующее значение в **`JsonElement`** (универсальный тип-«сырьё»), а потом по факту разобрать, что пришло.

## Базовый приём: декодировать в `JsonElement`

`JsonElement` — это абстрактный надтип, в который парсер уложит **любое** следующее значение: объект, массив, примитив или `null`. Тип определяется уже после чтения через `when`:

```kotlin
@OptIn(ExperimentalSerializationApi::class)
fun readNext(stream: InputStream) {
    val json = Json

    val el = json.decodeFromStream<JsonElement>(stream)   // читает ОДНО следующее значение, тип любой
    when (el) {
        is JsonObject -> {
            val user = json.decodeFromJsonElement<User>(el)        // дальше — в нужный класс
        }
        is JsonArray  -> {
            val users = json.decodeFromJsonElement<List<User>>(el)
        }
        is JsonPrimitive -> { /* строка/число/булево */ }
        JsonNull -> { /* null */ }
    }
}
```

Парсер сам находит конец значения (как обсуждали), поэтому ему не нужно заранее знать, объект там или массив — он смотрит на первый значащий символ (`{` или `[`) и читает до закрывающего.

## Поток из нескольких значений разного типа

Для последовательности смешанных значений — `decodeToSequence<JsonElement>`. **Важно явно указать `WHITESPACE_SEPARATED`**, а не оставлять `AUTO_DETECT`:

```kotlin
val seq = json.decodeToSequence<JsonElement>(stream, DecodeSequenceMode.WHITESPACE_SEPARATED)
for (el in seq) when (el) {
    is JsonObject -> { /* ... */ }
    is JsonArray  -> { /* ... */ }
    else -> { /* ... */ }
}
```

Почему не `AUTO_DETECT`: он решает по самому первому символу всего потока. Если первым придёт `[`, он сочтёт **весь поток одним массивом** (`ARRAY_WRAPPED`) и начнёт отдавать его элементы, а не «верхнеуровневые значения». Для потока вида `{...} [...] {...}` это сломает разбор. `WHITESPACE_SEPARATED` трактует каждое верхнеуровневое значение (и `{}`, и `[]`) как отдельный элемент.

## О чём помнить (компромисс)

- **`JsonElement` материализует значение целиком в дерево в памяти.** То есть для конкретного значения вы теряете «потоковость внутри него»: огромный массив будет полностью собран в `JsonArray`, прежде чем вы его получите. Чтение из сети остаётся инкрементным *между* значениями, но одно значение читается целиком.
- **Полиморфизм через дискриминатор тут не подходит.** Стандартный механизм sealed-классов с `"type": "..."` требует, чтобы значение было JSON-**объектом**, несущим поле-дискриминатор. Голый массив такого поля содержать не может, поэтому «объект ИЛИ массив» решается именно через `JsonElement`, а не через `@Serializable`-полиморфизм.
- Если значение — объект и тип внутри различается по полю, можно скомбинировать: сначала `JsonElement`/`JsonObject`, заглянуть в нужное поле (`el.jsonObject["type"]`), затем `decodeFromJsonElement<КонкретныйТип>(el)`.

## Чего библиотека НЕ даёт

Если нужно понять тип, **не материализуя значение** (например, заглянуть в первый токен гигантского массива и решать на лету) — публичного pull/token-API у kotlinx.serialization нет. Для такого сценария нужен Jackson (`JsonParser.nextToken()` / `currentToken()`), где можно посмотреть `START_OBJECT` vs `START_ARRAY` до полного чтения.

Если опишете формат потока (мешанина верхнеуровневых значений, NDJSON, или один тип с вариативной структурой) — подскажу точную комбинацию `decodeToSequence` + режим.
