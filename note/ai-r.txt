Разобрался с устройством хоста. Важный момент: на сборочном хосте под «версией Kotlin» скрываются **две независимые вещи**, и менять их нужно по-разному.

## 1. Что именно вы хотите обновить

| Что | Где задаётся | Чем определяется | Что компилирует |
|---|---|---|---|
| **Kotlin внутри Gradle** (1.9.22) | дистрибутивом Gradle 8.7 | версией самого Gradle | только build-скрипты `*.gradle.kts` |
| **Kotlin Gradle Plugin (KGP)** | в проекте (`plugins{}` / `libs.versions.toml` / `buildscript classpath`) | вами, в проекте | **код вашего приложения** ← это и есть «версия Kotlin для сборки» |

Embedded-Kotlin (1.9.22) напрямую не «ставится» — он меняется только вместе с апгрейдом дистрибутива Gradle. Для сборки приложения важна вторая строка — **KGP**, и именно её обычно имеют в виду.

## 2. Почему оффлайн это нетривиально

Когда вы пишете `kotlin("android") version "2.x"`, Gradle идёт в `mavenCentral()`/`google()` и тянет не один jar, а целое дерево: `kotlin-gradle-plugin`, `-api`, `kotlin-compiler-embeddable`, `kotlin-build-tools-impl`, `kotlin-daemon`, `kotlin-stdlib`, `kotlin-reflect` + транзитивные (coroutines, intellij-deps и т.д.). Руками их не собрать. Без сети хост это всё не скачает — поэтому артефакты надо **доставить заранее**.

Сейчас на хосте: в кэше только `kotlin-gradle-plugin/1.9.22`, локального `~/.m2` нет, init-скриптов нет, сеть наружу закрыта.

## 3. Рекомендуемый способ — локальный Maven-репозиторий + init-скрипт

**Шаг А — на машине С сетью** «прогреть» нужную версию и собрать из неё плоский репозиторий. Самый надёжный путь — реально собрать пустой проект на целевой версии Kotlin (тогда Gradle сам подтянет ровно нужный набор), затем выгрузить артефакты в каталог-репозиторий, например через init-скрипт `maven-publish`/копированием из `caches/modules-2`. Получите папку вида:

```
offline-repo/org/jetbrains/kotlin/kotlin-gradle-plugin/2.x/...
offline-repo/org/jetbrains/kotlin/kotlin-compiler-embeddable/2.x/...
...
```

**Шаг Б — перенести на хост** (сеть наружу закрыта, но SSH через jump-хост у нас есть):

```bash
scp -r offline-repo androidbuild:~/offline-repo
```

**Шаг В — заставить Gradle брать всё локально.** Создать `~/.gradle/init.d/offline.gradle` на хосте:

```groovy
allprojects {
    buildscript {
        repositories { clear(); maven { url "file:///home/builder/offline-repo" } }
    }
    repositories { clear(); maven { url "file:///home/builder/offline-repo" } }
}
settingsEvaluated { s ->
    s.pluginManagement.repositories { clear(); maven { url "file:///home/builder/offline-repo" } }
}
```

и собирать с `--offline` (или `org.gradle.offline=true` в `gradle.properties`).

**Шаг Г — поднять версию в проекте**: в `libs.versions.toml` / `plugins{}` указать новый KGP.

## 4. О совместимости (это часто и ломает апгрейд)

- Kotlin 2.x = новый компилятор **K2**; нужно проверить **AGP** (≥ 8.x) и саму версию **Gradle**.
- Если используется **KSP** — его версия жёстко привязана к версии Kotlin (формат `2.x.x-1.0.x`), её тоже надо положить в оффлайн-репо.
- Для Compose в Kotlin 2.0+ изменился способ подключения compose-compiler.
- Если хотите обновить и embedded-Kotlin — это апгрейд Gradle: положить локально `gradle-8.1x-bin.zip` и прописать `distributionUrl=file:...` в `gradle-wrapper.properties` (Gradle 8.11 несёт Kotlin 2.0.21).

## TLS

Тут важный момент: **отдельная библиотека для TLS как правило не нужна** — TLS встроен в JDK (пакеты `javax.net.ssl.*`: `SSLContext`, `SSLSocket`, `HttpsURLConnection`, и в Java 11+ — `java.net.http.HttpClient`). Kotlin использует тот же JVM-стек.

Дополнительные библиотеки берут, когда нужно больше, чем даёт JDK:

| Библиотека | Когда нужна |
|---|---|
| **встроенный JSSE (JDK)** | Базовый TLS, HTTPS — ничего ставить не надо |
| **BouncyCastle** (`org.bouncycastle:bcprov-jdk18on`, `bctls-jdk18on`) | Доп. алгоритмы, работа с сертификатами/ключами, кастомный TLS-провайдер |
| **Conscrypt** (`org.conscrypt:conscrypt-openjdk`) | TLS на базе BoringSSL/OpenSSL, новые версии TLS, производительность |
| **OkHttp** (`com.squareup.okhttp3:okhttp`) | HTTP-клиент с удобной настройкой TLS, certificate pinning |
| **Ktor** (`io.ktor:ktor-client-*` / `ktor-network-tls`) | HTTP-клиент/сервер на Kotlin, в т.ч. мультиплатформенный TLS |

## Короткий вывод

- **JSON**: нужна библиотека обязательно. Для Kotlin — `kotlinx-serialization-json`.
- **TLS**: на JVM **встроен в JDK**, отдельная библиотека требуется только для расширенных сценариев (BouncyCastle/Conscrypt) или если работаете через HTTP-клиент (OkHttp/Ktor), который сам инкапсулирует настройку TLS.

## Как проверять (команды)

Подключение:
```bash
ssh androidbuild2
```

**1. Runtime-библиотека где угодно в системе** (самое главное — нужны `core` и `json`):
```bash
find / \( -iname "kotlinx-serialization-core*.jar" \
         -o -iname "kotlinx-serialization-json*.jar" \) 2>/dev/null
```

**2. Кэш Gradle** (куда складываются зависимости проектов):
```bash
find ~/.gradle/caches -iname "*kotlinx-serialization*" 2>/dev/null
```

**3. Локальный репозиторий Maven:**
```bash
find ~/.m2 -iname "*kotlinx-serialization*" 2>/dev/null
# или по пути группы:
ls -R ~/.m2/repository/org/jetbrains/kotlinx/ 2>/dev/null | grep -i serial
```

**4. Что идёт в комплекте с самим Kotlin** (там лежит только compiler plugin, не runtime):
```bash
ls -1 /opt/kotlinc/lib/ | grep -i serial
```

Как отличать по имени:
- `kotlinx-serialization-**compiler-plugin**.jar` → плагин компилятора (есть всегда с kotlinc, для работы недостаточно).
- `kotlinx-serialization-**core**-<ver>.jar`, `kotlinx-serialization-**json**-<ver>.jar` → именно runtime, который и нужен для (де)сериализации.

Всё в одну строку (то, что запускал я):
```bash
ssh androidbuild2 'find / \( -iname "kotlinx-serialization-core*.jar" -o -iname "kotlinx-serialization-json*.jar" \) 2>/dev/null; \
  find ~/.gradle/caches ~/.m2 -iname "*kotlinx-serialization*" 2>/dev/null'
```
Пустой вывод первой части = runtime-библиотеки нет.
